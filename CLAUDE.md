## Project Overview

Gloomhaven Helper is an Android companion app for the Gloomhaven board game. It tracks teams, characters, scenarios, items, perks, personal quests, and active scenario gameplay using a local Room database.

./gradlew assembleRelease

## Architecture

The project follows Clean Architecture with four layers:

- **Presentation** (`screens/`): Jetpack Compose UI + ViewModels with `StateFlow<UiState>`
- **Domain** (`domain/`): Use cases for business logic, entity models, error types
- **Data** (`data/`): Repositories coordinating between domain and persistence, mappers
- **Persistence** (`bd/`): Room database with DAOs, entities, type converters, and JSON fillers

### Key Patterns

- **Reactive data flow**: Room DAOs return `Flow<Entity>` → Repositories transform to `Flow<DomainModel>` → ViewModels emit `StateFlow<UiState>` → Compose collects via `collectAsStateWithLifecycle()`
- **Navigation**: Type-safe routes defined in `navigation/GlHelperScreens.kt` using `@Serializable` data classes/objects
- **DI**: Hilt with `@Singleton` scope for database, DAOs, and repositories; modules in `di/`
- **Character classes**: Stored as `CharacterClassType` enum, not in database. `TeamCharacterClassBd` tracks unlocked classes per team

### Database

- Room database configured in `bd/GlHelperDatabase.kt`
- Schema exports to `app/schemas/` for migration validation
- Migrations in `bd/migrations/` - add to `ALL_MIGRATIONS` array
- Initial data loaded from JSON via fillers in `bd/filler/json/`, orchestrated by `bd/filler/DatabaseFiller.kt` (gated by a `filler_version` pref)
- Type converters in `bd/typeconverters/` for JSON serialization of complex types

When changing database schema:
1. Increment version in `GlHelperDatabase.kt`
2. Create migration object (e.g., `MIGRATION_1_2`) in `bd/migrations/`
3. Add migration to `ALL_MIGRATIONS` array
4. Build to export new schema

Note: the app is pre-release. Schema v3 intentionally relies on `fallbackToDestructiveMigrationFrom(false, 3)` to wipe and reseed rather than carrying a real `MIGRATION_2_3` — clearing stale data is a requirement, not an oversight. Write a real migration only once the app has shipped.

### Localization (multi-language data)

Catalog content is split into locale-invariant base tables and per-locale translation tables:

- **Base tables** (`GoodBd`, `MonsterBd`, `ScenarioBd`, `AchievementBd`, `PerkBd`, `PersonalQuestBd`, `LocationBd`, `MonsterStatsBd`) hold language-neutral data and are never modified by translation. Translatable entities use **stable string slugs** (or display numbers) as primary keys rather than autogenerated ints, so saved progress and game logic survive re-seeding.
- **Translation tables** (`*TranslationsBd` / `*TranslateBd` / `MonsterTextStatsBd`) hold `(slug-or-number, locale, text…)` rows, keyed `(entityKey, locale)`, with `ForeignKey(onDelete = CASCADE)` back to the base table (note: `GoodTranslationsBd` currently has no FK).
- **Resolution happens in SQL**, not in memory: DAO queries `LEFT JOIN` the translation table twice and pick the text with `COALESCE(targetLocale, defaultLocale, <stable id>)`. The final `COALESCE` arg is the entity's slug/number so a missing translation renders a stable identifier, never a placeholder string.
- **Locale flows reactively**: `LocaleRepository.observeLocale` combines system + app locale; use cases `flatMapLatest` on it so a language change re-runs queries. DAOs take `targetLocale` + `defaultLocale` params; `LocaleRepository.DEFAULT_LOCALE` is `"en"`.
- **Fallback subquery pattern**: when a query returns the default-locale row only if no target-locale row exists, it uses a *correlated* `NOT EXISTS` against the **same** translation table (see `PerksDao.getPerksByCharacterClass`). Copy this shape exactly — referencing the wrong table or dropping the correlation/parentheses silently returns wrong-locale or wrong-entity rows.
- **Assets**: per-pack, per-locale JSON under `app/src/main/assets/data/<pack>/<locale>/` (e.g. `main/en/monsters.json`), with locale-invariant data in `data/<pack>/`. `DatabaseFiller` seeds each pack (`main`, `forgottenCircles`) for every locale.

### Key Entities

- `TeamBd` - Team with achievements, packs, reputation, prosperity
- `CharacterBd` - Character stats, level, experience, gold
- `ScenarioBd` / `TeamScenarioBd` - Scenarios and team progress
- `MonsterBd` / `MonsterStatsBd` / `MonsterAbilityCardBd` - Monster data (keyed by `slug`); `MonsterTextStatsBd` holds localized stat text
- `ScenarioGameStateBd` - Active scenario state (monsters, rounds, cards)
- `GoodBd` / `PerkBd` / `PersonalQuestBd` / `LocationBd` - Items, perks, quests, scenario locations
- `*TranslationsBd` / `*TranslateBd` - Per-locale text for the above base tables (see Localization)
